This page contains several section about radio usage and help. Please check the menu for the different sections.

To use the radio fully you will have to obtain an API key. This can be achieved via the miku bot that sits on the #radio IRC channel (but can be reached over private message too). You will have to associate your nick first (after authenticating via Satsuki). Check .associate for more details.

Make sure to read the Privacy considerations section to learn how we handle your data.

Stream information

You can use the Web site to playback and control the radio. Alternatively, you can use these direct links to use your preferred playback application.

If you want to list yourself as a listener, you can use one of your API keys for identifying. For example: https://radio.animebits.moe/stream/stream128.ogg?apikey=AbCdEfGhI_0-123456789.

If the place where you connect from blocks multimedia, you can try to add bypass/ to the url like this: https://radio.animebits.moe/stream/bypass/stream128.ogg. This will change the Content Type to text which bypasses some company proxies.

Links are only available under HTTPs. Some bitrates/formats have higher delay than others.

Codec Container Links
AAC ADTS 256kbit/s 128kbit/s 64kbit/s
Opus Ogg 256kbit/s 128kbit/s 64kbit/s
MP3 MP3 192kbit/s (.m3u)
FLAC FLAC 16-bit

Note that the streamer will take the original media and convert it realtime to the streamed codec. Most existing media is at least FLAC-16bit, although some are only available in lossy format.
When this media played, a transcode will occur. If streaming the FLAC link, an upconversion will occur so you might notice different quality compared to other songs.

#radio miku bot Help & Commands

You can use miku over the #radio channel or via private message. Some queue-related commands might be limited to channel only.

By default, all users identified via Satsuki have standard rights. If you are not identified, ask in the channel to be granted rights. There are higher right levels for moderation purposes that have extra abilities or commands, those are not listed here.

If you want to identify and secure your nick, you will have to run .associate then .genkey to get an API key.

.skip

Skips the currently playing track. If you are the only listener on the radio, you should have a control on the web interface to do this as well. Only works in channel.

.np

Shows the track that is now playing

.listeners, .l

Shows the current listeners

.queue, .q

Shows the current queue, up to three top entries. If empty, it will try to find the next random track that will play.

.history, .h

Shows the last few recently played tracks.

.related

Shows the current random playback mode.

.top [day|week|month] [player]

Shows the top#6 tracks from the last week/month/day, or a player link with the top 20 if specified.

.qpop

Removes the last entry in the queue. Only works in channel.

.qshift

Removes the first entry in the queue. Only works in channel.

.favorite [hash], .fav [hash]

Favorites the now playing track. If [hash] is specified, favorite that track instead. If you have a different radio nick than your IRC nick, please use .associate first

.unfavorite [hash], .unfav [hash]

Unfavorites the now playing track. If [hash] is specified, unfavorite that track instead. If you have a different radio nick than your IRC nick, please use .associate first

.associate <nick>

Associates radio <nick> with your IRC nick for favorite-related commands and authentication. You need to be authed via Satsuki. Check .genkey to learn how to create an API key afterwards.

.nick <nick>

Changes radio nick to <nick>

.unassociate

Unassociates radio nick -> IRC nick relation for favorite-related commands. Only works in channel.

.genkey <name>

Creates or regenerates a new API key that you can use to authenticate on the web interface or the API (put this on an Authorization header when requesting things). You can have multiple API keys, each with a different name. You have to use .associate first

NOTE: This command only works over private message.

.lskeys

Lists all of your API keys.

NOTE: This command only works over private message.

.rmkey <name>

Removes one of your API keys.

NOTE: This command only works over private message.

.search <query>, .s <query>

Does a track search and displays the top three results. You can then send the number (just 1 or 3, or multiple like 1,3,5) of the result(s) to queue them. <query> can use multiple selectors (title, album, artist, fav, duration, hash, audio, lyrics, tag, favcount, playcount, tagcount) to reduce the number of results and find the track you want, for example: .search album:"naruto" title:"come back" duration>100

Advanced queries are also supported, using currently playing context ($album or $artist) to refer to current playing track details, quotes, equality operators, or AND / OR / NOT (-) / parenthesis. By default every new term you add uses AND (with higher precedence), but you can specify OR instead (note they must be uppercase).

For example: favcount>3 (artist=shibayan OR artist="ELECTRO.MUSTER") @mns -"el dorado" #remix

All searches are case insensitive. When using the equality operator, you can include % as a wildcard.

You can use .more to expand the list of results afterwards. Be nice and do this over private message if you want to go through a lot of them.

For a more in-depth look at this rich syntax, look at the stewdio-api search code.

.qrand <query>

Does a track search and queues a random result. Same syntax for <query> as .search. Can also specify these commands to queue up to three tracks at once: .qrand1 <query>, .qrand2 <query>, .qrand3 <query>

.qfav [query]

Queues a random favorite. [query] can be specified to limit which favorites to randomly pick, same syntax as for .search. Can also specify these commands to queue up to three tracks at once: .qfav1 [query], .qfav2 [query], .qfav3 [query]

.req <query/hash>

Same as .search, but if there is only a single result it will queue it directly.

.request <link>

Requests an album/artist/collage/series to be added to the radio. Only AnimeBytes / JPopSuki / RED links allowed.

.fetch <link>

Requests an existing direct media file, pulls metadata out, and tries to find it on the radio. Multiple results displayed.

Also works with these radio sites (link stream or page directly instead of file): Stewdio Radio System, r/a/dio, edenofthewest, JpopSuki, gensokyoradio, animenfo, listen.moe

Out-Of-Band Player

The radio has ways to playback your favorited tracks or other queries like albums or single songs without having to put them live on the radio. Most of these require your browser to playback the original file format, but thanks to modern technology we are able to implement JavaScript decoding of most file formats even if your browser does not support them.

This player also supports timed lyrics on either LRC or ASS format. If you have timed lyrics for any track/album, ask on the IRC channel.

For any of these links you can append /m3u or /pls to generate a playlist file. It will also work with a dot instead of a forward slash.

Favorites player

You can access your (or other user) favorites using this link format https://radio.animebits.moe/player/favorites/<nick> where <nick> is a valid user nick.

Album Group player

You can list an album(s) using the original group / torrent / series / artist id from a site. Valid tags are abg (AnimeBytes group id), abt (AnimeBytes torrent id), abs (AnimeBytes series group id), aba (AnimeBytes artist id), jpsg (JPopSuki group id), jpst (JPopSuki torrent id) and others.

Example: https://radio.animebits.moe/player/abg/8915 would link the AnimeBytes group id 8915 (in this format, abg/8915) which corresponds with a final group url like this https://animebytes.tv/torrents2.php?id=8915, which in this case corresponds to Mirage by Mell.

Album Name player

You can list an album(s) using the direct name of it, if you can't find a tagged version of it. Example: https://radio.animebits.moe/player/album/RETRO FUTURE GIRLS where in this case RETRO FUTURE GIRLS is the album name.

Search player

In case you want to show a longer list or a subset search, you can use this player with any query that works via miku / radio player. Example: https://radio.animebits.moe/player/search/favcount>0 artist=primary where in this case we search for anything with more than 0 favorites from artist equal to primary.

Tracklist player

In case you want to show a single song or create a small playlist, you pass the song hash(es) as given by miku / API. Example: https://radio.animebits.moe/player/hash/36730236b14c6f715a114845bcb4c373 where in this case we find the song Clockup Flowers from MÁGICO CATÁSTROFE.

Shorter hashes are allowed, or playlists like this https://radio.animebits.moe/player/hash/56e9b1fe,36730236,2aa2cb1b,501b6504,bffe5221,cdd15308,32329b02,b8e84d89,781344a8,1bf36780,45648ccb,c9abfdce,d45d5ab0,8830d4a8,71d06a4b,487c23cf,0b602f46,6f3953d1,5f63e1f3,c7386fa6.

Exporting files offline

You can write your own tools that use the API to fetch files. Otherwise, we provide a bash script that lets you query the radio database and fetch selected tracks. A use case of this would be exporting your favorites for offline playback. Do note that you are limited to a few number of concurrent connections, and there might be daily limits in the future.

In doubt, you can use the API to extract tag information, like torrent IDs that you can fetch directly from the original site.

If you want to use this bash script, you can grab radio-downloader.sh over here, and remember to chmod +x radio.downloader.sh afterwards. Do note you will have to edit the file to include an API_KEY on it.

Some dependencies might need to be installed, in this case curl and jq (JSON command line parser). On Debian/Ubuntu you can install them via sudo apt install curl jq.

Usage: ./radio-downloader.sh <query>, where <query> is any normal search query. It can be set with and without quotes, but remember to escape special characters like > < @ # and similar.

After running the query, the script will ask you which tracks to fetch. You can configure some settings on the top next to API_KEY, and present your own download path over environment variables. Defaults to current directory.

Importing favorites

You might have a large collection of favorite tracks already. If you want to add these to your radio favorites, we got you covered.

Head over radio.animebits.moe/import and you can use that to find matches locally without sending any data to the search API except metadata

You need to be authenticated on main radio page to use this service.

UserScripts

Some userscripts are provided that add extra radio functionality to some other related sites.

You will need an UserScript manager like TamperMonkey, GreaseMonkey, VioletMonkey or similar to use these.

anime(bits) extension for AnimeBytes: Adds several direct album links for player to torrent groups, artists, collages, series, forums with, and other search links for finding related tracks.

anime(bits) cover extension for AnimeBytes: Allows finding new covers directly from torrent group, and other utilities to find further data.

API documentation

This radio features an extensive API, which is the main way to interact with the radio, and even the landing page uses it. If you don't have an API key, you will be limited to which methods you can use to limit leakage of user data.

The main API URL sits under https://radio.animebits.moe/api/. There is no versioning, and details could change (although usually not in a breaking way). This could change in the future but will be announced on any relevant channels or threads.

Track/song record format

Anywhere you query or receive song information, you will get back JSON that looks similar to this, depending if you are authenticated or not.

{
   "id": 283694,
   "title": "Magia",
   "path": "/mnt/nas-ceph/JapanMusic/Albums/after eden (2011)/03 - Magia.flac",
   "duration": 309,
   "hash": "b76f73be23ec611efb77500ad21919a9",
   "play_count": 8,
   "audio_hash": "513f2cac72429681a3d93134209a283e",
   "album": "After Eden",
   "artist": "Kalafina",
   "cover": 24368,
   "status": "active",
   "favored_by": [ "mns", "theblackhawk", hummer12007", "amm" ],
   "tags": [ "abt-35494", "pop", "abg-11986", "aba-424", "catalog-secl-1014", "catalog-sexx01066b00z" ]
}

You should familiarize yourself with this format as you will see it everywhere. Some details for your handling of this data:

  • id is an internal field. You will not use it to refer to anything in this API and could change.
  • hash can be considered your unique identifier. Most endpoints that act or show information on a song will use this to refer to them. This is an MD5 digest across the entire file, tags and all. If this hash mathces a file you have, they are the same.
  • audio_hash is an MD5 digest of raw audio data applied over the decoded audio as signed raw PCM16. This is useful for finding exact audio dupes (with altered metadata).Most endpoints that act or show information on a song will use this to refer to them.
  • title, album and artist can change when files are tagged or newer metadata is fetched.
  • path relates to the full storage path the file is stored on. This could change.
  • cover gets you the album/song cover image.
  • duration notes the length of the track in seconds. On some files this duration could be corrupt due to bad metadata. If you use it for finding matches, a threshold of 1-2s around this number is recommended for better matching.
  • status shows whether a track exists on disk / can play or not. Should you find a track not marked active, disregard the record, but it may become active in the future.

Authentication

Unless explicitly said so, assume all API endpoints require authentication. You can obtain any amount of API keys through miku.

Currently, there are three methods of authenticating, although some are not applicable on some endpoints:

  • Via the apikey request URL parameter. Example: https://radio.animebits.moe/api/info/b76f73be23ec611efb77500ad21919a9?apikey=AbCdEfGhI_0-123456789
  • Via the radio-apikey request cookie header. Example: Cookie: radio-apikey=AbCdEfGhI_0-123456789
  • Via the Authorization request header.
    This can be any of these ways of sending it: raw header (example: Authorization: AbCdEfGhI_0-123456789), filled username as API key and empty password, or filled password as API key and empty username.

If denied, you will get a 401 response code. You can generate new keys or remove/regenerate old keys via miku anytime.

Realtime WebSocket information

If you are interested in live playback or events as they happen, this is the method you should use. Scraping the other API endpoints to check for changes in them is heavily discouraged.

There are two endpoints, one without authentication and with limited events and fields broadcasted, and other with authentication that contains the full set of events and fields.

/events/all

Once connected, you will receive an initial dump of information related to the events being listened, then any new changes will be broadcasted directly to you. Every new event sent is JSON-encoded.

These are the current events being broadcasted. The field type specifies which event you are receiving, while the field data contains the information.

  • playing gets sent to inform of the current track being played. Contains a song record with some additional fields like play time start.
  • listeners gets sent to inform of the current number of listeners, and who is listening.
  • queue gets sent to inform of queue additions / removals. It contains several fields like an action (with values such as initial, add or remove), a queue_id, and a song record. You also receive these when tracks move to next one.
  • favorite gets sent to inform of players adding or removing tracks to their favorites. It contains an action (with values such as add or remove) and a song record.
  • random gets sent to inform of the next random track that will be played if queue is empty. Contains a song record.

/events/basic

This endpoint does not require authentication.

Once connected, you will receive an initial dump of information related to the events being listened, then any new changes will be broadcasted directly to you. Every new event sent is JSON-encoded.

These are the current events being broadcasted. They have been cleaned of favorites, path, tags and other identifying information. The field type specifies which event you are receiving, while the field data contains the information.

  • playing gets sent to inform of the current track being played. Contains a song record with some additional fields like play time start.
  • listeners gets sent to inform of the current number of listeners.
  • queue gets sent to inform of queue additions / removals. It contains several fields like an action (with values such as initial, add or remove), a queue_id, and a song record. You also receive these when tracks move to next one.
  • random gets sent to inform of the next random track that will be played if queue is empty. Contains a song record.

Privacy considerations

Although this is an user run site, we take privacy as a main concern. We offer different ways to use the radio, and different methods to prevent drive-by extraction of user data.

By default, any unauthenticated API call or client has all the identifiable information removed from the response. A list of some of these actions, not complete:

  • Listener count does not contain any names and just number of guests.
  • Song favorite details are removed from all data.
  • Hashes, tags and paths are removed from all data.
  • They do not get favorite broadcasts.
  • They are not allowed to use any API method except the /events/basic WebSocket (which prunes this information), the /cover method (which shows them a cover), and /download rate-limited method (to use OoB player links).
  • Users not authenticated via Satsuki on site will not be able to use miku commands.

All connections to the site require SNI and a modern browser / client that supports TLSv1.2 or TLSv1.3, and only ECDHE Forward Secrecy modern ciphers are enabled (with server-preferred order). Certificates are auto-renewed by LetsEncrypt (4096bits RSA currently), and secure connections are enforced via HSTS. DNSSEC is enabled on our domain and DANE TLSA records set. You are connecting directly to us. Security updates are installed automatically.

All resources used on site / web are stored locally, and requests are not sent to any other servers.

When you connect to the radio stream, you will be sending us your IP address (it's how the Internet works). We don't keep logs with any information that can identify the user behind the request (no URL query, no IPs, no Cookies), and these logs are not stored on disk. But don't trust us, use a VPN.

You have the possibility of listening to the radio while unauthenticated. This means you are just a "guest", not someone with a nick. You are still able to use the #radio IRC channel or miku over PM, this will not give us any information about your connection to the radio site.

When you enter #radio or use miku, you are letting us see your AnimeBytes username and class (you need to be auth'd via Satsuki to use commands). If you use one of your API keys on the web interface / stream link, this creates a link between IP <-> AnimeBytes user. While we do not store or show any IP information, this could be something you don't want to do.

When you use the favorite features you are letting other users see your chosen nick in the songs you favorite, and allow searching for other songs you like. If you do not want this but still want to use this feature, we recommend you use a random made-up nick.

If you want to delete all data stored for your account (nick, favorite list, API keys) you can contact us on the #radio channel. If you want to rename your nick, we can also help you there.

TL;DR: Use a VPN, we don't want to know what your IP is. Do not authenticate on Web UI / stream and use just IRC control features if you don't trust us. Use a made-up nick if you don't want other users seeing what you like.

Source code

Most of the parts used to build this site were written by others originally, some were patched, others were written from scratch. Listed below are the distinct parts that compose the radio, and if able, a location with the currently available source code.

  • radio-docker: Merges all projects together in a single platform, and containerizes it via Docker and docker-compose. Used for testing and site itself in the future.
    Source code is available at github.com/WeebDataHoarder/radio-docker.
  • radio-web: Radio website. Though mainly static, Out of Band player has some extra PHP code to run the backend search and matching.
    Source code is available at github.com/WeebDataHoarder/radio-web.
  • stewdio-api: Main music database, and API interaction system written in Python. Forked from original stewdio-api. Adds extra features (covers, database indexes, extra search, and others) and mainly removes user management.
    Source code is available at github.com/WeebDataHoarder/stewdio-api.
  • kawa: Radio live streamer and queue management written in Rust. Fork with some changes from original.
    Source code is available at github.com/WeebDataHoarder/kawa.
  • radio-matcher: Metadata matcher and finder. Finds and collects metadata from other sources, and creates tags. All this is then pushed to main music database for normal use. Mainly a collection of PHP scripts and produced database.
    Source code is not available at this moment :(
  • miku: IRC bot. Handles user management, provides rich search and other more useful features. Mainly PHP.
    Source code is available at github.com/WeebDataHoarder/radio-ircbot.
